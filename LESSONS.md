# Lessons Learned

## HeXO Game AI Optimization
*   **Move Pruning Issue:** The initial HeXO AI logic aggressively downsampled candidate moves (randomly sampling 50 out of 200+ legal moves) to combat high branching factors. This random sampling caused the AI to frequently miss critical blocking and winning moves since it couldn't reliably see its most important immediate options.
*   **Smart Selection:** Replacing random downsampling with localized pruning (restricting candidate moves to distance <= 2 from any existing stone) naturally reduced the branching factor down to ~20-30 highly relevant moves without sacrificing crucial local play analysis. This ensures the AI continues to build contiguous structures effectively.

## PyTorch Model Training & RTX Ada Optimization
*   **Architecture Choice:** For HeXO's hexagonal axial-coordinate grid, employing scalable 2-Dimensional Convolutional Networks over a standardized matrix (`21x21`) resolves the enormous CPU latency historically created by Graph node instantiations. Models infer lightning fast globally instead of dragging.
*   **Ada Lovelace Hardware Optimization:** To make the network properly ingest training data efficiently on an RTX 2000 Ada equivalent platform, two mechanisms were needed: `torch.autocast('cuda')` together with `torch.cuda.amp.GradScaler()` to shrink the footprint using Automatic Mixed Precision (BFloat16/Float16), alongside setting `torch.set_float32_matmul_precision('high')`. This forcefully engages the TF32 tensor cores which are heavily under-utilized on consumer/laptop 4000-series cards if left standard.

## Python Engine Scalability & MCTS Bottlenecks
*   **Deepcopy vs Clone:** MCTS relies on rapidly executing thousands of state projections. Relying on `copy.deepcopy(state)` on Python nested dataclass dictionaries generated an enormous 10-100x overhead penalty. Utilizing a custom `clone()` method that exclusively employs shallow list/reference copying methodologies neutralized this latency natively.
*   **O(1) Radius Caching:** Dynamically evaluating a 289-hex bounding box radially across every single stone per end-turn `_update_reachable_cache()` caused computational loops to surge aggressively into $O(N \cdot 289)$ ranges, crashing training SPS by >50%. A `pending_cache_updates` structure allowed the engine to systematically parse exclusively new stones to update the bounds instead, guaranteeing a perfect $O(1)$ loop operation regardless of the board state footprint.
*   **Immutable Tuple Hashing vs Dynamic Sorting:** Applying `str(sorted(...))` mathematically forced MCTS state hashing arrays to iteratively invoke dynamic sorting logic frames linearly upon string concatenation internally. Re-mapping board dictionaries into `frozenset` coupled alongside numerical tuples natively shifted evaluation off pure python logic frames and directly onto Python's fully-internal C-hashing core structures natively, preserving inference efficiency.

## GUI Integration and User Experience
*   **Model Fallback Mechanism:** When linking an AI inference setup to a GUI frontend interface, providing an automatic and seamless fallback logic natively inside the integration wrapper (e.g., using `os.path.exists()` onto model `.pth` binaries combined with an alternative internal Random AI approach) prevents frustrating setup and testing crashes inherently, granting instantaneous access whether or not extended hardware optimization runs have been concluded gracefully.
*   **State Transparency:** Adding a visible marker to clarify model engagement (e.g., UI tags rendering `[AI - Model]` versus `[AI - Random]`) provides the user with an immediately observable method of confirming integration integrity seamlessly.
*   **Contiguous Element Rendering:** In dense grids (like continuous hex-tile placements), uniformly filled colors merge adjacent pieces together visually, neutralizing depth perception and masking structural sequence length. Trimming or overlapping a standard stroke order natively over the grid with a dark `width` border explicitly segments identical objects, drastically simplifying pattern evaluations (e.g. counting to 6) for human players inherently.
*   **Persistent Artifact Protection:** When training models locally (e.g., generating `hexo_model.pth`), tracking these binaries in Git risks accidental overwrites during cross-device pulls. Implementing a strict `.gitignore` for binary model artifacts ensures that local training progress remains persistent and sovereign regardless of repository synchronization events across different hardware.

## Training Script Integrity and Performance
*   **Functional Imports:** Standardizing on `import torch.nn.functional as F` across all training scripts (even if primarily using the `nn` module) prevents runtime `NameErrors` when calculating complex losses (like Cross-Entropy or MSE) inside the training loop. Functional interfaces are often more ergonomic for custom loss logic than their module counterparts.
*   **Mixed Precision Alignment:** When using `torch.amp.autocast`, ensure that BOTH the model inference and the loss calculations are wrapped within the autocast context to maintain gradient precision. Failure to wrap the loss calculation can lead to numerical instability or `NaN` losses during the backpropagation step on RTX Ada GPUs.

## Performance Scaling & SPS Maintenance
*   **Incremental Hashing vs. Frozenset:** Hashing the entire board on every MCTS step using `frozenset(board.items())` causes exponential slowdown as games progress. Implementing an incremental XOR-base hash (`_board_hash`) that updates only when stones are placed reduces hashing overhead from $O(N)$ to $O(1)$, stabilizing SPS.
*   **Engine Boundary Pruning:** Without a boundary radius, the `_update_reachable_cache` loop can expand the "legal moves" set into thousands of hexes that fall outside the NN's receptive field. Hard-coding a `boundary_radius` matching the training grid prevents this "off-board bloat" from dragging down search performance.
*   **Multiprocessing Resource Hygiene:** In long training runs, failing to explicitly shut down `multiprocessing.Manager()` processes between epochs can lead to resource exhaustion and context-switching overhead. Always wrap Managers in `with` contexts to ensure cleanup after each epoch's self-play phase.
*   **Vectorized MCTS Selection:** Pure-Python loops over hundreds of candidate moves for UCB calculation (Selection phase) are highly inefficient. Using NumPy to calculate UCB scores for all legal moves simultaneously provides a significant throughput boost, especially on high-branching factor games like HeXO.
